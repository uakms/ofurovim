(エラー)メッセージの說明

*message.txt*   For Vim バージョン 8.1.  Last change: 2019 May 18


		  VIMリファレンスマニュアル    by Bram Moolenaar

このファイルには Vim が表示するメッセージとエラーメッセージのリストが、アルファベット順で紹介してある。メッセージの意味がわからなかつた時にはこれを使つて調べることができる。だけど完璧ではない。

1. 古いメッセージ|:messages|
2. エラーメッセージ|error-messages|
3. メッセージ|messages|

1. 古いメッセージ

既に發生したメッセージを見るのに ":messages" コマンドが使用できる。これはメッセージが上書きされてしまつたり、スクロールで流れていつてしまつた時にとりわけ便利である。これは ’shortmess’ オプションに依存する。

:messages

すべてのメッセージを表示する。

:{count}messages

新しい方から {count} 件のメッセージを表示する。

:messages clear

すべてのメッセージを削除する。

:{count}messages clear

新しい方から {count} 件を殘して、メッセージを削除する。

見ることのできるメッセージ數は、tiny バージョンでは 20 に、それ以外のバージョンでは 200 に固定されてゐる。

コマンド "g<" を使ふと前のコマンドの出力の最後のページを見ることができる。hit-enter プロンプトで誤つて <Space> を押してしまつたときにこれは便利である。このコマンドを使ふと hit-enter プロンプトまで戾り、そこでスクロールバックができる。

Note:
出力を表示中に "q" を押してそれ以降の出力表示をさせなかつた場合は、"g<" を使つたときもそこまでしか表示されない。

他のコマンドが出力を行ふと、前のコマンドの出力は消去される。"g<" の出力はリダイレクトされない。

飜譯されたメッセージを使つてゐるなら、誰がそのメッセージや飜譯をメンテナンスしてゐるかが最初の行に表示される。閒違ひを見つけたときはそこを見てメンテナーと聯絡をとることができる。

特定の(エラー)メッセージに關するヘルプを見つけたいときは、メッセージの最初に表示される ID を使ふ。例へば: 

E72: Close error on swap file

または飜譯版なら: 

E72: Errore durante chiusura swap file

といふメッセージに關するヘルプを見つけるには: 

:help E72

とすればよい。面倒ならシフトキーを使はないでもできる: 

:help e72

2. エラーメッセージ

エラーメッセージが表示されたが、それを讀む前に消えてしまつた場合には、次のコマンドでもう一度見ることができる: 

:echo errmsg

もしくは最近のメッセージのリストを見る: 

:messages

詳しくは上の ‘:messages‘ を參照。

メッセージのリスト

Add to read buffer
makemap: Illegal mode
Cannot create BalloonEval with both message and callback
Hangul automata ERROR
block was not locked
Didn't get block nr {N}?
ml_upd_block0(): Didn't get block 0??
pointer block id wrong {N}
Updated too many blocks?
get_varp ERROR
u_undo: line numbers wrong
undo list corrupt
undo line missing
ml_get: cannot find line {N}
cannot find line {N}
line number out of range: {N} past the end
line count wrong in block {N}
Internal error
Internal error: {function}
fatal error in cs_manage_matches
Invalid count for del_bytes(): {N}

これは內部エラーである。これを再現できる場合はバグレポートを送つてください。|bugs| 

ATTENTION
Found a swap file by the name ...

|ATTENTION| を參照。 

Buffer {N} not found

あなたが要求したバッファは存在しない。これはマークを含んでゐたり、その他の方法で參照されてゐるバッファを完全に削除 (wiped out) したときも起こりうる。|:bwipeout| 

Buffer with this name already exists

同じ名前のバッファを2つ持つことはできない。 

Close error on swap file

編輯中のテキストのコピーを保存するスワップファイル |swap-file| が適切にクローズできなかつた。たいていは無害。 

Command too recursive

Ex コマンドが Ex コマンドを實行する Ex コマンドを實行...(以下略) となつたとき起こる。この制限は、200 または ’maxfuncdepth’ のどちらか大きいほうである。それ以上多いときはおそらく無限ループである。おそらく |:execute| か |:source| コマンドが關係してゐるだらう。 

Cannot allocate color {name}

色名 {name} が不明。ほとんどのシステム上で利用可能な色のリストが |gui-colors| にあるので參照。 

Cannot allocate colormap entry, some colors may be incorrect

Vim に必要な色數が使用できないことを意味してゐる。Vimは終了しないが、一部の色が正しく表示されない。多くの色を使つてゐさうなアプリケーションを止めるか、もしくは Vim を起動した後で起動してみると良い。 ブラウザーは多くの色を浪費することで知られてゐる。netscape の場合は、獨自の colormap を使ふやうに指定することで囘避できる: 

netscape -install

もしくは色數の制限をしてしまふ (64 くらゐが良いらしい): 

netscape -ncols 64

同じことを Xdefaults を使つてやるなら: 

Netscape*installColormap: Yes

もしくは 

Netscape*maxImageColors:  64

Cannot expand wildcards

ファイル名がをかしな文字の組み合はせを含んでゐる。そのため Vim はワイルドカードを展開しようとしたが、失敗した。これはマッチするファイル名が見つからなかつたといふことではなく、パターンが不正であることを意味してゐる。 

Cannot go back to previous directory

ファイル名を展開中に、Vim が以前使はれてゐたディレクトリに戾ることに失敗した。かうなると、使はれてゐる全てのファイル名が無效になるかもしれない。ユーザーはカレントディレクトリに實行パーミッションを持つてゐる必要がある。 

Cannot open "{filename}" for writing
Can’t open file for writing

なんらかの理由によつて、あなたが書かうとしてゐるファイルは作成または上書きができない。理由は、そのディレクトリに書き込みパーミッションを持つてゐないか、ファイル名が無效であることかもしれない。 

Can’t open linked file for writing

あなたは上書きできないファイルに書き込みをしようとしてゐて、そのファイルはリンク(ハードリンクまたはシンボリックリンク)である。そのリンクまたはファイルのあるディレクトリが書き込み可能なら、それでも書き込むことは可能だが、あなたがそのリンクを削除してその場所にファイルを書き込みたいのか、ファイルそのものを削除してその場所に新しいファイルを書き込みたいのか Vim にはわからない。もし本當にファイルをこの名前で書き込みたいのなら、手動でリンクまたはファイルを削除するか、Vim がそれを上書きできるやうにパーミッションを變更しなければならない。 

Cannot change read-only variable "{name}"

函數の引數 |a:var| または Vim 內部變數 |v:var| に値を代入しようとした。これらはリードオンリーである。 

Cannot unload last buffer

Vim は常に少なくとも 1 つのバッファをロードしてゐなければならない。さうでなければウィンドウに表示するものが何もなくなつてしまふ。 

Can’t open errorfile <filename>

":make" や ":grep" コマンドを使つた時: エラーメッセージや grep の結果を開くことができなかつた。これには幾つかの原因が考へられる:

  • ’shellredir’ に閒違つた値が入つてゐる。
  • シェルがディレクトリを變更してしまつたために、エラーファイルが違ふディレクトリに作られてしまつた。’makeef’ を變更することで修正できるが、make コマンドは誤つたディレクトリで實行されてしまふことになる。
  • ’makeef’ に閒違つた値が入つてゐる。
  • ’grepprg’ や ’makeprg’ を實行できない。これは必ず檢出できるとは限らない(特に Windows では)。$PATH をチェックすること。 
Can’t open file C:\TEMP\VIoD243.TMP

MS-Windows において、外部コマンドの出力を讀み込まうとしたが、コマンドが正常に動作しなかつたときに現れる。これには多くの原因がありうる。’shell’, ’shellquote’, ’shellxquote’, ’shellslash’ やこれらに關聯したオプションをチェックすること。外部コマンドが見つからなかつたといふこともありうる。その場合の特別なエラーメッセージは用意されてゐない。 

Command not allowed from exrc/vimrc in current dir or tag search

いくつかのコマンドはセキュリティ上の理由のために許可されてゐない。それらのコマンドはたいていカレントディレクトリの .exrc や .vimrc または tags ファイルから來る。’secure’ も參照。 

Command too complex

マッピングが非常に長いコマンド文字列に展開されてしまつた。マッピングが閒接的に自分自身を呼ぶことによつて起こつた可能性がある。

變換エラー

ファイル書き込み時に "CONVERSION ERROR" といふ文字が現れたら、これは內部で使はれてゐる UTF-8 からファイルのフォーマットに變換するときにいくつかのビットが失はれたことを意味する。このファイルには unmodified のマークがされない。情報の缺落が氣になるなら、バッファ中の文字を扱へるやうに ’fileencoding’ オプションを他の値にし、再度書き込みを行ふこと。氣にしないならバッファを破毀したり ’modified’ オプションをリセットしてもよい。バックアップファイルがある場合、’writebackup’ または ’backup’ が設定されてゐると削除されないので、變更を破毀したい場合は元に戾すことができる。 

Could not rename swap file

ファイル名が變更されたとき、Vim は |swap-file| も同樣にリネームしようとする。これが失敗し、以前のスワップファイルが引き續き使はれる。多くの場合は無害。 

Damaged match string
Corrupted regexp program

Vim 內部でなんらかの不具合が起こり、正規表現が壞れてしまつた。この問題を再現できるなら報告してください。|bugs| 

Error writing to "{filename}"
Error closing "{filename}"
Error reading "{filename}"

これは Vim がファイルをリネームしようとしたが、ファイル名の單純な變更ができなかつたときに起こる。するとファイルはコピーされるが、どういふわけかこれが失敗した。結果として、元のファイルとコピー先のファイルの兩方が存在するが、コピー先のファイルは不完全な可能性がある。 

Vim: Error reading input, exiting...

これは入力が求められてゐるときにタイプされた文字が讀み込めなかつた場合に起こる。Vim はどうにもならなくなり、終了するしかない。これは標準入力と標準出力の兩方がリダイレクトされてゐて、かつ Vim を終了させないスクリプトを實行してゐるときに起こりうる。 

Error while reading errorfile

エラーファイルの讀み込みに失敗した。これは、エラーメッセージが認識できないことによつて引き起こされるわけではない。 

Error while writing

ファイルの書き込みが正常に完了しなかつた。そのファイルはおそらく不完全である。 

File exists (add ! to override)
"{filename}" exists (add ! to override)

閒違つてファイルを上書きしてしまはないやうに保護されてゐる。どうしても上書きしたいのなら、同じコマンドの直後に "!" を付けて使用する。例: 

:w /tmp/test

これをこう變へる: 

:w! /tmp/test

Swap file exists: {filename} (:silent! overrides)

Vim で編輯してゐる最中のファイルは上書きしないやうになつてゐる。このエラーはスワップファイルが存在するのに ":w! filename" を實行したときに發生する。

  • もしそのスワップファイルが昔のクラッシュした編輯セッションで作られたものならこのスワップファイルは消したいだらう。{filename} を編輯するとそのスワップファイルについての情報が得られる。
  • 强制的に保存したいならコマンドの前に ":silent!" をつける。例: 
    :silent! w! /tmp/test

    既存のファイルを上書きするために既に ! をつけてゐるので、この特別なコマンドが必要になる。 

File is loaded in another buffer

あなたは別のバッファで使はれてゐる名前でファイルを書き込まうとしてゐる。さうすると同じファイルの 2 つの版ができてしまふ。 

File not written: Writing is disabled by ’write’ option

’write’ オプションがオフである。これはファイルを書き込まうとする全てのコマンドが生成するメッセージである。コマンドライン引數で |-m| を指定することで同じメッセージを發生することができる。":set write" で ’write’ オプションを設定することができる。 

GUI cannot be used: Not enabled at compile time

GUI のためのコードを含んでゐない Vim のバージョンを實行してゐる。從つて "gvim" と ":gui" は機能しない。 

Invalid scroll size

これは ’scroll’, ’scrolljump’, ’scrolloff’ オプションに無效な値を設定することによつて引き起こされる。 

"{filename}" is a directory

あなたは既存のディレクトリの名前でファイルを書き込まうとした。これは不可能である。ファイル名を追加する必要があるだらう。 

Mark has invalid line number

存在しない行番號を持つマークを使用した。別のファイルのマークを持つてゐるときか、別のプログラムがファイルからラインを消してしまつた場合に起こり得る。 

Missing {.
Missing }.

ファイル名の中に {} を使つてゐるが、{ に對應する } がない、またはその逆。これは {foo,bar} のやうに使はねばならない。これは "foo" と "bar" にマッチする。 

ml_get: invalid lnum: {number}

Vim の內部エラー。どうしたらこれを再現できるかを發見して、バグレポートを送つてくれないだらうか |bugreport.vim|。 

{number} more files to edit

引數リストの最後の項目を編輯してゐないのに終了しようとしてゐる。これはまだ編輯すべきファイルがあるのに誤つて終了するのを防止する。|argument-list| を參照。もし本當に終了したいのなら、もう 1 度繰り返せば終了できる。 

No alternate file
No alternate file name to substitute for ’#’

オルタネートファイルがまだ定義されてゐない。|alternate-file| 參照。 

No file name

現在のバッファに名前が存在してゐない。ファイルに保存するためには ":w filename" を使ふ。もしくは ":file fname" でバッファに名前を付ける。 

No file name for buffer {number}

變更されたバッファの 1 つがファイル名を持つてゐない。そのためそのファイルを書き込めない。そのバッファにファイル名を與へる必要がある: 

:buffer {number}
:file {filename}

No previous substitute regular expression

パターンに文字 ’~’ を使用すると、以前に ":substitute" コマンドで使はれたパターンに置き換へられる。そのやうなコマンドがまだ使はれてゐない時には、これは失敗する。|/~| 參照。また、これは ":s/pat/%/" を使つたときにも起こる。この "%" は以前の置換で使はれた置換後文字列を意味する。 

No previous regular expression

空の檢索パターンを使用したときには、以前に使はれた檢索パターンが再度使用される。しかし、以前に檢索が行はれてゐないのならば無理な話だ。 

No such abbreviation

":unabbreviate" コマンドを、存在してゐない略語を引數にして實行した。":cunabbrev"、":iunabbrev" などのバリエーションも同じメッセージを與へる。後ろに空白がついてゐないかチェックすること。 

/dev/dsp: No such file or directory

Gnome サポートつきの GTK GUI でのみ表示される。Gnome がオーディオデバイスを使はうとしたが、利用可能でない。このエラーは無視してよい。 

No such mapping

":unmap" コマンドを、存在してゐないマップを引數にして實行した。":cunmap"、":unmap!" などの、このコマンドのバリエーションも同じメッセージを生成する。いくつかのヒント:

  • 後ろに空白がついてゐないかチェックすること。
  • バッファローカルなマッピングについては ":unmap <buffer>" とする必要がある。|:map-<buffer>| 
No write since last change (add ! to override)
No write since last change for buffer {N} (add ! to override)

變更のあるファイルを抛棄 (|abandon|) しようとした。Vim は仕事 (ファイル) を失つてしまふ危險から守つてくれる。變更されたファイルを ":w" で保存するか、變更を失つたとしても本當に抛棄 (|abandon|) するかのどちらかである。破毀するには、使つたコマンドに ’!’ を付けて實行すればよい。例: 

:e other_file

これをこう變へる: 

:e! other_file

No write since last change for buffer "{name}"

變更されたバッファがあるのに Vim を終了しようとしたとき表示される。變更されたバッファを |:w| で書き込むか、":qa!" などのバッファを强制的に破毀するコマンドを使ふかしなければならない。保存しておきたい變更を破毀しないやうに注意すること。特に ’hidden’ をセットしてゐるときは、バッファのことを忘れてゐるかもしれない。 

[No write since last change]

このメッセージは、少なくとも 1 つのバッファが變更されてゐるときにシェルコマンドを實行したときに現れる。このメッセージを表示させなくするには、オプション ’warn’ をオフにする。 

Null argument

Vim 內部で何か不具合が起こり、NULL ポインタが現れてしまつた。もしこの問題を再現できるなら報告してください。|bugs| 

Out of memory!
Out of memory!  (allocating {number} bytes)
Cannot allocate any buffer, exiting...
Cannot allocate buffer, using other one...

あ～あ… 何か厄介なことをしたか、別のプログラムがメモリを喰らひ盡くしたか。氣をつけて。Vim はメモリ不足の狀況といふのは、想定してゐない。まづ、變更したファイルが保存されてゐるか確認しよう。それからメモリを節約する方法を考へよう。安全に使ひたいのならば、一度 Vim を終了して、再度起動しませう。

バッファは一部のみメモリに保持されるので、大きなファイルを編輯するときでもメモリ不足になることはほとんどない。アンドゥ情報は完全にメモリに保持されるが、次の方法でその量を減らすことができる:

  • ’undolevels’ を小さい値に設定する。-1 に設定するとアンドゥ機能を完全に無效にできる。すべての行を變更するやうな場合に役立つ。
  • ’undoreload’ をゼロに設定して無效化する。 
Pattern too long

int 型が 16 ビットのシステム上で起こる: コンパイルされた正規表現パターンの長さが 65000 文字以上である。もつと短いパターンを試してみること。

また、ルールのオフセットが利用可能な空閒に收まらない場合にも起こる。もつとシンプルなパターンを試してみること。 

’readonly’ option is set (add ! to override)

讀み込み專用とマークされたファイルに書き込まうとした。本當にファイルを書き込んでしまふには、’readonly’ オプションをリセットするか、保存に使用したコマンドに ’!’ を付けて實行し直すかのどちらか。例: 

:w

これはこう變へる: 

:w!

Read error in swap file
Seek error in swap file read
Oops, lost the swap file!!!

Vim が |swap-file| からテキストを讀み込まうとしたが、何かまづいことが起こつた。關係するバッファのテキストが破壞されてゐるかもしれない。バッファを書き込む前に注意深くチェックすること。違ひをチェックするために、別のファイルに書き出すといいかもしれない。 

Recursive use of :normal too deep

":normal" コマンドの引數で再び ":normal" コマンドを使ひ、再歸的になつてゐる。これは ’maxmapdepth’ で回數が制限されてゐる。この例ではどうやつてそれを引き起こすかを示してゐる: 

:map gq :normal gq<CR>

"gq" とタイプすれば、このマッピングが實行され、その結果再び "gq" が呼び出される。 

Scripts nested too deep

スクリプトはコマンドライン引數の "-s" や、":source" コマンドで讀み込むことができる。スクリプトは、他のスクリプトファイルを讀み込むことができる。これは 14 段階まで續けることができる。それ以上に入れ子にした場合には、Vim は再歸的なループがあると假定し、このエラーメッセージを出して讀み込みを止める。 

Sorry, the command is not available in this version

使用中の Vim では利用できないコマンドを使はうとした。Vim のコンパイル時に、いくつもの機能を有效／無效にすることができる。これは Vim をどれだけ大きくするかの決定と OS に依存する。いつどの機能が利用できるのかを見るには |+feature-list| を參照。|:version| コマンドを使ふとVimがどの機能つきでコンパイルされたかを表示できる。 

Swap file already exists (symlink attack?)

Vim がスワップファイルを開かうとしたが、それが既に存在するかその場所にシンボリックリンクがあつたとき表示される。これは本來起こつてはならない。なぜなら Vim はすでにそのファイルが存在しないことをチェックしたからである。他の誰かが同じファイルをちやうど同じ瞬閒に開いたか(これはほぼありえない)、誰かが symlink 攻擊をしようとしてゐるかである (/tmpにあるファイルを編輯中か、’directory’ が "/tmp" で始まつてゐる場合に起こりうる。’directory’ を "/tmp" 以下に設定すべきではない)。 

Tags file not sorted: {file name}

Vim (および Vi) は tags ファイルが ASCII 順でソートされてゐることを期待する。さうすれば線形探索よりもかなり高速な二分探索が可能になるからである。もしも tags ファイルがソートされてゐなかつたら、|'tagbsearch'| オプションをリセットする必要がある。

このメッセージは Vim が tag を探してゐる時に問題があると表示される。tags ファイルがソートされてゐないにもかかはらず、このメッセージが表示されないことも時々ある。 

Too many different highlighting attributes in use

Vim は約 223 種類までの强調表示を扱ふことができる。この限界に達してしまつたといふことは、異なる引數に對して |:highlight| コマンドを使ひすぎたことを意味する。":highlight link" はカウントされない。 

Too many file names

ファイル名を展開して、マッチが 1 個以上見つかつた。直前に使はれたコマンドに對してはマッチが 1 つしか許されない。 

Unable to open swap file for "{filename}", recovery impossible

Vim がスワップファイルを作成できなかつた。ファイルを編輯することはできるが、もし Vim が不意に終了したら、變更は失はれる。また、巨大なファイルを編輯するときはメモリを多く消費するかもしれない。このエラーを囘避するために ’directory’ オプションを變更するとよいかもしれない。|swap-file| を參照。 

Use ! to write partial buffer

範圍選擇をしてバッファの一部を書き込むとき、元のファイルを上書きしようとすることはあまりない。これはおそらく誤りである (例へばビジュアルモードがアクティブなときに ":w" を使つた)。それゆゑ Vim はコマンドの後に ! をつけることを要求する。例: ":3,10w!"。 

Warning: Cannot convert string "<Key>Escape,_Key_Cancel" to type VirtualBinding

起動時にこのやうなメッセージが現れた場合、これは Vim の問題ではなく、X11 の設定が閒違つてゐる。この問題を解決するためのヒントがここにある:

http://groups.yahoo.com/group/solarisonintel/message/12179.

[この URL はもはや有效ではない] 

Warning: Changing a readonly file

ファイルが讀み込み專用なのに、變更を加へようとしてゐる。このメッセージを避けるには、自動コマンドイベント |FileChangedRO| を使ふことができる(自動コマンドは ’readonly’ オプションをリセットしなければならない)。ファイルへの變更を完全に禁止するには ’modifiable’ を參照。

このメッセージは ’readonly’ がセットされて以降最初の變更に對してのみ現れる。 

Warning: File "{filename}" has been created after editing started

存在しないファイルの編輯をしてゐたはずが、今は同名のファイルが存在する。現在開いてゐるファイルの編輯を續けるか、新しく作成されたファイルを開くか決めなければならない。’buftype’ が空でないときはこのメッセージは表示されない。 

Warning: File "{filename}" has changed since editing started

ファイルが編輯を始めた時とは違ふタイムスタンプを持つてゐて、その內容が變更されてゐる (より正確には: 現在のオプション設定と自動コマンドでそのファイルを讀み直した結果、異なるテキストになつてしまつたとき) 多分他のプログラムがそのファイルを書き換へたのだらう。何が起こつたのか究明し、どつちの版を殘したいのか決斷する必要がある。これを自動的に行はせたいのなら ’autoread’ オプションをセットすること。’buftype’ が空でないならばこのメッセージは表示されない。

何も閒違つたことをしてゐないのに、このメッセージを受け取る可能性が 1 つ考へられる: 夏時閒が始まる日に Windows でファイルを保存した場合である。この場合、次の方法のうちどれかで修正できる:

  • 次の行を autoexec.bat に加へる: 
    SET TZ=-1

    タイムゾーンに "-1" の修正を施す。

  • "automatically adjust clock for daylight saving changes" を無效にする。
  • 單に、次の日にファイルを再び書き直す。または時計を次の日にセットし、ファイルを 2 回書き込み、時計を戾す。

もし W11 が常に表示される場合、"Acronis Active Protection" を無效にするか、Vim を信賴できるサービス/アプリケーションとして登錄する必要があるかもしれない。 

Warning: File "{filename}" has changed and the buffer was changed in Vim as well

上のものと同樣だが、Vim 內のバッファも同樣に變更された。Vim 中の版とディスク上の版のどちらを殘すのか決めなければならない。’buftype’ が空でない場合、このメッセージは表示されない。 

Warning: Mode of file "{filename}" has changed since editing started

バッファのタイムスタンプが變更され、中身は同じだがモード (パーミッション) が變更されたとき。これは通常、讀み込み專用ビットをリセットするやうなバージョン管理システムからファイルをチェックアウトしたときに起こる。ファイルを讀み直すのが安全である。自動的にファイルを讀み直させるやうにするには ’autoread’ をセットすること。 

File "{filename}" no longer available

編輯を始めてゐるファイルが消え失せたか、もはやアクセスできない。變更が失はれるのを避けるため、バッファが書き込まれたか確認すること。’buftype’ が空でないときはこのメッセージは表示されない。 

Warning: List of file names overflow

とてつもなくたくさんのバッファを使つてゐる。かうなると 2 つのバッファに同じ番號が割り當てられるやうになり、樣々な問題を引き起こす。Vim を終了し、再起動するとよいかもしれない。 

Buffer cannot be registered

メモリ不足になつたかバッファ番號が重複した。W14 の後に發生するかもしれない。バッファ檢索は常には機能しないので、Vim を再起動したはうが良い。 

Seek error in swap file write
Write error in swap file

これはたいていディスクが一杯のときに起こる。Vim が |swap-file| にテキストを書き込むことができなかつた。直接的には有害ではないが、Vim が不意に終了したとき、テキストが失はれ、復活の可能性もなくなつてしまふかもしれない。この問題が持續する場合は、Vim がメモリを使ひ果たしてゐるのかもしれない。 

Xlib: connection to "<machine-name:0.0" refused by server

Vim が X サーバーへ接續を試みたが、接續を確立できなかつた場合にこれが起こる。X サーバーへの接續は、タイトルと xterm のクリップボードをサポートするために必要とされる。殘念なことに、|+xterm_clipboard| と |+X11| の機能を無效にしない限り、このエラーメッセージは避けることはできない。 

\\ should be followed by /, ? or &

コマンドラインがバックスラッシュで始まつてゐるか、コマンドの範圍のをかしな場所にバックスラッシュが含まれてゐる。これはしばしばコマンドラインの行繼續が無效にされてゐるため引き起こされる。行繼續を有效にするには ’cpoptions’ から ’C’ フラグを除くか、または ":set nocp" とすること。 

Argument required

引數を必要とする Ex コマンドに對し、引數を與へないで實行したとき起こる。 

Invalid argument
Invalid argument: {arg}
Duplicate argument: {arg}

Ex コマンドまたは函數が實行されたが、無效な引數が指定された。 

Trailing characters

引數をつけてはならない Ex コマンドに引數が指定された。 

No ! allowed
Don’t panic!

"!" をつけてはならないExコマンドに "!" をつけた。 

No range allowed

範圍指定を許可してゐない Ex コマンドに範圍を指定した。|cmdline-ranges| を參照。 

Can’t create file {filename}
Can’t get temp file name

Vim が一時ファイルを作成できなかつた。 

Can’t open file {filename}
Can’t read file {filename}

Vim が一時ファイルを讀み込めなかつた。特に Windows では、cmd.exe での特殊文字のエスケープがうまくできない場合に發生する。パッチ 7.3.443 でエスケープ方法が變更された。|system()| に與へるすべてのシェル引數に |shellescape()| を使つてみるとよい。あるいは、明示的に ^ でエスケープする方法もある。’shellxquote’ と ’shellxescape’ も參照。 

Ambiguous use of user-defined command

名前の先頭部分が一致するユーザー定義コマンドが 2 つあり、そのうちの 1 つを實行しようとコマンドライン補完をした。|user-cmd-ambiguous| を參照。例: 

:command MyCommand1 echo "one"
:command MyCommand2 echo "two"
:MyCommand

Not an editor command

Ex コマンドでもユーザー定義コマンドでもないコマンドを實行しようとした。 

コマンドテーブルはアップデートされる必要がある。’make cmdidxs’ を實行する。

これはソースコードを變更した時と src/ex_cmds.h にコマンドを追加した時にのみ生じる。このルックアップテーブルは次を實行してアップデートされる必要がある: 

make cmdidxs

3. メッセージ

Vim がくれるさまざまなメッセージの(不完全な)槪觀である: 

Press ENTER or type command to continue

このメッセージは讀んでもらひたい何かがあつて、スクリーンの再描畫が中斷されてゐるときに與へられる:

  • 外部コマンドの實行直後 (例へば ":!ls" と "=")。
  • ウィンドウの幅よりも長い何かがステータスラインに表示されたか、’showcmd’ か ’ruler’ の出力にまで達してしまつた。
  • -> スクリーンを再描畫し再開するには、<Enter> か <Space> (他の用途に使はれてゐないことが條件)を押す。
  • -> 何かコマンドを使ふために、":" か他のノーマルモードコマンドのキャラクタを叩く。
  • -> ’k’, <Up>, ’u’, ’b’, ’g’ を押すとメッセージが上にスクロールする。これは |more-prompt| と同樣に働く。’compatible’ がオフで ’more’ がオンのときだけ有效である。
  • -> ’compatible’ がオフで ’more’ がオンのとき、メッセージが畫面上端を越えてスクロールした場合、’j’, ’f’, ’d’, <Down> は無視される。これは ’j’, ’f’ を押しつぱなしにしたときメッセージが消えてしまふのを避けるためである。
  • -> <C-Y> を押し、モードレスセレクションをクリップボードレジスタにコピー(ヤンク)する。
  • -> メニューを使ふ。コマンドラインモード用に定義されてゐる文字が使はれる。
  • -> ’mouse’ が ’r’ フラグを含んでゐる場合は、マウスの左クリックが <Space> を押すのと同樣になる。ただし ’r’ フラグを含んでゐるとテキストを選擇できなくなる。
  • -> GUI を使つてゐるときは、最後の行で左クリックすると <Space> を押すのと同樣になる。

もし誤つて <Enter> や <Space> を押してしまつて、テキストを表示し直したいときは |g<| を使ふ。これは ’more’ がオンのときだけ有效である。

hit-enter のプロンプトの回數を減らすには:

  • ’cmdheight’ を 2 かそれ以上に設定する。
  • ’shortmess’ にフラグを追加する。
  • ’showcmd’ と ’ruler’ の兩方か、もしくはどちらかをリセットする。

スクリプトが hit-enter プロンプトを出すが、なぜそれが出るのかわからない場合は變數 |v:scrollstart| が便利かもしれない。

’mouse’ も參照。hit-enter メッセージは |hl-Question| グループで强調される。 

-- More --
-- More -- SPACE/d/j: screen/page/line down, b/u/k: up, q: quit

このメッセージはスクリーンがメッセージで埋め盡くされた時に表示される。’more’ オプションがオンの時だけ與へられる。|hl-MoreMsg| グループで强調される。

種類效果
<CR> or <NL> or j or <Down>1 行進める
d次の半ページ
<Space> or f or <PageDown>次のページ (1畫面)
G最後へ (hit-enter プロンプトまで)
<BS> or k or <Up>1 行戾る
u前の半ページ
b or <PageUp>前のページ
g最初へ戾る
q, <Esc> or CTRL-Cリスト表示を止める
:リスト表示を止めコマンドラインに入る
<C-Y>モードレスセレクションをクリップボード (レジスタ "* と "+) にヤンク (コピー) する。
{メニューの項目}コマンドラインでメニューが定義されてゐること
<LeftMouse>次のページ (*)

その他のキーは、これらのキーの意味が表示される。

(*)

左クリックは次のときだけ機能する:

  • GUI 使用時:スクリーンの最後の行で
  • ’mouse’ が ’r’ を含んでゐるとき(ただしこのときテキスト選擇はできない)

Note:
タイプされたキーはターミナルから直接得られ、マップされず typeahead は無視される。

コマンド |g<| を使ふと前のコマンドの出力の最後のページを見ることができる。hit-enter プロンプトで誤つて <Space> を押してしまつたときにこれは便利である。
